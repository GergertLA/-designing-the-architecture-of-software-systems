# Лабораторная работа №1

**Тема:** Формулирование требований к программной системе.
**Цель работы:** Научиться анализировать поставленную задачу, формулировать функциональные и нефункциональные требования к проектируемой системе.

### 1. Перечень заинтересованных лиц (стейкхолдеров)

**Руководитель проекта** - определяет цели разработки системы, принимает ключевые решения, предоставляет финансирование.

**Разработчики системы** - команда специалистов (backend, frontend, ML-инженеры, DevOps), отвечающих за проектирование, реализацию, тестирование и поддержку всех компонентов системы.

**Трейдеры** - основные пользователи системы. Используют результаты анализа тональности и новостного влияния для оперативного принятия торговых решений.

### 2. Перечень функциональных требований

#### 1. Сбор новостных данных
**1.1.** Система должна автоматически собирать новости из заранее заданных источников (криптовалютные СМИ, финансовые порталы, социальные сети).

**1.2.** Система должна выполнять извлечение и структурирование метаданных каждой новости: дата и время публикации, источник, автор (если доступно), заголовок, основной текст, ссылка на оригинал.

**1.3.** Система должна сохранять сырые (необработанные) новостные данные в базе данных для последующей обработки и анализа.

#### 2. Предобработка текстовых данных
**2.1.** Система должна очищать текст новостей от HTML-разметки, спецсимволов, эмодзи и сервисной информации.

**2.2.** Система должна выполнять нормализацию текста: токенизацию, приведение к единому регистру, стандартизацию формата.

**2.3.** Система должна удалять стоп-слова, выполнять лемматизацию/стемминг с целью улучшения качества дальнейшего анализа.

#### 3. Анализ тональности и тематическая классификация
**3.1.** Система должна классифицировать тональность каждой новости как позитивную, негативную или нейтральную.

**3.2.** Система должна определять степень уверенности (confidence).

**3.3.** Система должна выделять ключевые слова и фразы, оказывающие влияние на оценку тональности.

**3.4.** Система должна классифицировать новости по тематическим категориям (технологии, макроэкономика, хакерские атаки, политические события и др.).

#### 4. Анализ влияния на цену Bitcoin:
**4.1.** Система должна собирать и хранить исторические данные цены Bitcoin и объёмы торгов.

**4.2.** Система должна сопоставлять временные метки новостей с временным рядом цены для проведения анализа.

**4.3.** Система должна выполнять корреляционный и статистический анализ взаимосвязи между тональностью/темами новостей и изменениями цены Bitcoin.

**4.4.** Система должна рассчитывать количественные метрики влияния новостей (например, изменение цены через 5 мин, 30 мин и 1 час после события).

#### 5. Визуализация результатов:
**5.1.** Система должна отображать интерфейс для совместной визуализации динамики sentiment-анализа новостей и ценовых колебаний Bitcoin.

**5.2.** Система должна визуализировать статистику по источникам новостей: частота публикаций, вклад в позитивный/негативный фон, историческое влияние.

#### 6. Управление системой:
**6.1.** Система должна предоставлять возможность настройки параметров сбора новостей: выбор источников, периодичность обновления, фильтры по темам или языкам.

**6.2.** Система должна обеспечивать обучение, переобучение и обновление моделей машинного обучения по мере накопления новых данных.

**6.3.** Система должна предоставлять инструменты мониторинга: статус сборщиков данных, корректность работы моделей, протоколирование ошибок и логирование операций.

**6.4.** Система должна отправлять уведомления о критических ошибках или остановке модулей.

### 4. Перечень сделанных предположений

#### 1. Источники данных
- предполагается доступ к бесплатным или платным API новостных агрегаторов;
- исторические данные о цене Bitcoin доступны через открытые API (CoinGecko, Binance и др.);
- новостные источники включают основные финансовые СМИ и специализированные крипто-издания.

#### 2. Технические предположения
- система будет разрабатываться на Python с использованием библиотек для ML и NLP;
- для хранения данных будет использоваться реляционная база данных;
- доступ к вычислительным ресурсам для обучения моделей достаточен для обработки больших объемов текста.

#### 3. Функциональные предположения
- тональность новостей является значимым фактором для прогнозирования ценовых движений;
- временная задержка между публикацией новости и рыночной реакцией составляет от нескольких минут до часов;
- качество анализа тональности будет достаточным для практического применения.

#### 4. Бизнес-предположения
- пользователи имеют базовое понимание принципов работы криптовалютных рынков;
- система будет использоваться для поддержки принятия решений, а не для полностью автоматической торговли;
- существует рыночный спрос на инструменты анализа новостного фона для Bitcoin.

### 5. Перечень нефункциональных требований

#### 1. Производительность:
**1.1. время обработки одной новости не должно превышать 5 секунд;**

**1.2.** время отклика пользовательского интерфейса не более 2 секунд.

#### 2. Надежность:
**2.1.** доступность системы 99% времени в рабочее время;

**2.2.** автоматическое восстановление после сбоев в течение 15 минут;

**2.3. резервное копирование данных ежедневно.**

#### 3. Безопасность:
**3.1.** защита персональных данных пользователей;

**3.2.** аутентификация пользователей при доступе к системе;

**3.3. шифрование конфиденциальных данных при хранении и передаче.**

#### 4. Совместимость:
**4.1. поддержка веб-браузеров последних версий (Chrome, Firefox, Safari);**

**4.2.** REST API для интеграции с внешними системами;

**4.3.** поддержка форматов экспорта данных (CSV, JSON).

#### 5. Удобство использования:
**5.1. интуитивно понятный пользовательский интерфейс;**

**5.2.** подробная документация для пользователей.
